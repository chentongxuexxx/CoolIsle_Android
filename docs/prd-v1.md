# 缓岛 V1.0 增量产品需求文档

**文档版本**：V1.0-Inc
**撰写日期**：2025年
**项目代号**：Huandao (com.huandao.app)
**技术栈**：Kotlin + Android Native + Room + Material Design 3
**最低 SDK**：24 | **编译 SDK**：34

---

## 1. V1.0 产品目标

> 构建一个 **离线优先、温暖陪伴** 的消费决策缓冲工具，用户可将冲动消费意向加入「冷静池」获得 72 小时缓冲期，在非评判的环境下做出更清醒的购买决策。

---

## 2. P0 用户故事（必做，7 条）

| # | 用户故事 | 验收条件 |
|---|---------|---------|
| **US-01** | 作为用户，我希望能**快速添加一个消费念头**到冷静池（名称+金额），操作不超过 3 步、不超过 10 秒，以便在冲动时第一时间"放下"而非"下单"。 | 点击添加 → 输入名称和金额 → 点击「先放一放」，流程完成；金额数字键盘自动弹出；成功后卡片动画落入列表。 |
| **US-02** | 作为用户，我希望能**在冷静池列表看到所有缓冲中的念头**，每条显示商品名、金额、已缓冲时长、状态色标，以便一眼了解当前有哪些待决策项。 | 列表以卡片流展示；卡片左侧竖线按状态着色（缓冲中 #E8913A / 已到期 #4CAF50 / 已决策 #9E9E9E）；空状态展示温暖插画+引导文案。 |
| **US-03** | 作为用户，我希望能**左滑删除或标记完成某个念头**（决定购买 / 不再考虑），操作直觉化且两个选项权重相等、无评判色彩。 | 左滑露出两个等大按钮：「我决定买了」/「不再考虑」；确认后记录变为终态，7 天后归档至历史；toast 文案温暖中性（"决定了就好""这个念头放下了"）。 |
| **US-04** | 作为用户，我希望能**为一个消费念头打上情绪标签**（如"压力大""被种草了""奖励自己"），选择是可选的、多选的（最多 3 个），以便日后发现自己的情绪-消费模式。 | 添加面板展示 8 个预设标签（水平滚动）；支持 +自定义标签（2-6 字）；标签选中/未选中视觉分明；不选标签也能完成添加。 |
| **US-05** | 作为用户，我希望能**看到每个念头已缓冲了多久**，在添加时**选择冷静期时长**（默认 72 小时，可选 24h/72h/7 天），冷静期到期时收到**一条温和提醒**，而非强制催促。 | 卡片显示"已缓冲 X 小时/天"；倒计时归零后状态变为「到期待决策」；Push 通知文案："关于[商品名]，你现在怎么想？"——不含催促语气。 |
| **US-06** | 作为用户，我希望能看到**价格的中性换算**（如"≈ 每天 11 元——供参考""≈ 3 顿工作餐——供参考"），纯粹提供信息锚点，不做价值判断。 | 输入金额后实时展示 1-2 条换算结果；所有换算以"——供参考"结尾；换算维度可选（日薪/实物/时间等价）；未设置日薪时展示引导但不强制。 |
| **US-07** | 作为用户，我希望能**在冷静期到期后获得结构化决策引导**——回答几个温和的开放式问题来帮助自己理清想法，但最终决策权完全在自己手中。 | 到期后进入决策引导页；问题如"现在的你还想买吗？"；三个等权按钮：「我还是想买」/「好像没那么想了」/「再等等」；选择「再等等」则重置冷静期。 |

## 3. P1 用户故事（选做，5 条）

| # | 用户故事 | 验收条件 |
|---|---------|---------|
| **US-08** | 作为长期用户，我希望能**查看消费洞察报告**（周摘要 + 月度深度报告），了解自己的情绪-消费关联模式（如"压力大时常看服饰"），而非仅仅看到花了多少钱。 | 周报：每周一推送，3-4 个核心数据点（本周添加数、放弃率、情绪标签分布），30 秒可浏览完；月报：包含情绪-品类热力图、时段分布柱状图、趋势变化等，底部可生成分享卡片。 |
| **US-09** | 作为用户，我希望能通过**成就系统获得正向激励**——奖励"觉察"行为（如记录情绪、完成冷静期、连续使用），而非奖励"省钱金额"。 | 首批 15-20 个成就（入门/情绪/冷静/持续使用四类）；解锁时顶部 toast 弹入 3 秒后消失；成就墙页面网格展示，未解锁显示"???"灰色态；所有成就条件与金额无关。 |
| **US-10** | 作为用户，我希望能**生成并分享脱敏卡片**——分享我的"清醒时刻"或情绪洞察，但卡片上**不出现任何金额、商品名**等个人财务信息，只传递"缓冲与觉察"的理念。 | 四种卡片类型（清醒时刻卡/情绪觉察卡/成就解锁卡/缓冲中卡）；分享前预览可编辑文案；支持保存相册和系统分享。 |
| **US-11** | 作为用户，我希望能有一个**个人中心**，查看累计使用数据（念头数/决策数/使用天数），管理偏好设置（默认冷静期、换算偏好、提醒开关），以及**导出或删除我的所有数据**。 | 数据统计摘要准确；导出为 JSON+CSV 打包 ZIP，通过系统分享发出；删除数据需二次确认（输入"确认删除"）。 |
| **US-12** | 作为用户，我希望能**收到克制的通知提醒**（冷静期到期、周报、成就解锁），每天最多 1 条非用户触发的推送，且所有通知类型可独立开关，免打扰时段 22:00-08:00 默认静默。 | 冷静期到期只发 1 次；周报每周一 10:00 发；成就解锁实时但低优先级；通知首开不主动请求权限，第一次添加冷静池后温和询问。 |

## 4. UI 交互流程

### 4.1 页面层级与导航

```
BottomNavigation (3 Tab)
├── Tab 1: 冷静池 (首页)
│   ├── 冷静池列表（默认）
│   ├── 历史记录（筛选：已购买/已放弃）
│   └── 搜索/筛选
│
├── Tab 2: 洞察 (P1)
│   ├── 本周摘要
│   ├── 月度报告
│   └── 情绪-消费图谱
│
└── Tab 3: 个人中心 (P1)
    ├── 用户卡片 & 数据摘要
    ├── 缓冲偏好设置
    ├── 数据管理（导出/清除）
    ├── 提醒设置
    ├── 我的成就 (P1)
    └── 关于 & 反馈
```

### 4.3 添加面板布局要点

- **类型**：BottomSheet（底部半屏浮层），可下滑收起
- **区域分布**：
  - 商品名称：全宽文本输入
  - 金额 + 品类：同行，金额左 40% + 品类右 60%
  - 情绪标签：水平滚动行，8 个预设 + 自定义入口
  - 冷静期选择：4 个 chip 选项行
  - 主按钮：「先放一放」全宽 #E8913A 圆角 12px
- **换算实时展示**：输入金额后，下方展开换算结果区域

### 4.4 冷静池卡片规格

- 左侧 2px 竖线：状态色标
- 商品名：16px #333333
- 金额：18px #E8913A 加粗
- 情绪标签：小圆角标签 #FFF0E0 底 #E8913A 字
- 已缓冲时长：12px #999999 右下角
- 换算结果：14px #999999，"——供参考" 结尾

## 5. 数据模型概要

### 5.1 核心实体 (Room Entities)

```
┌──────────────────┐       ┌──────────────────────┐
│   CoolPoolItem   │       │     EmotionTag       │
├──────────────────┤       ├──────────────────────┤
│ id: String (PK)  │  N:M  │ id: String (PK)      │
│ title: String    │◄─────►│ name: String         │
│ price: Double?   │       │ emoji: String?       │
│ category: String?│       │ isPredefined: Bool   │
│ coolPeriodHours: │       │ colorHex: String?    │
│   Int (def=72)   │       │ createdAt: Long?     │
│ createdAt: Long  │       └──────────────────────┘
│ expiresAt: Long  │
│ status: String   │       ┌──────────────────────┐
│ (cooling/expired │       │  CoolPoolItemTagCrossRef │
│  /decided_buy/   │       ├──────────────────────┤
│  /decided_pass)  │       │ itemId: String (FK)  │
│ decidedAt: Long? │       │ tagId: String (FK)   │
│ decisionReason:  │       └──────────────────────┘
│   String?        │
│ sourceUrl: String?│
│ archivedAt: Long? │
└──────────────────┘
```

### 5.2 Room 数据库表

**cool_pool_items**

| 列名 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | UUID |
| title | TEXT NOT NULL | 商品名称，最长 50 字 |
| price | REAL | 预估金额，可为空 |
| category | TEXT | 品类：服饰/数码/美食/家居/美妆/娱乐/其他 |
| cool_period_hours | INTEGER NOT NULL DEFAULT 72 | 冷静期时长（小时） |
| created_at | INTEGER NOT NULL | 创建时间戳 (ms) |
| expires_at | INTEGER NOT NULL | 到期时间戳 (ms) |
| status | TEXT NOT NULL DEFAULT 'cooling' | cooling / expired / decided_buy / decided_pass |
| decided_at | INTEGER | 决策时间戳 |
| decision_reason | TEXT | 决策理由（可选） |
| source_url | TEXT | 商品来源链接（可选） |
| archived_at | INTEGER | 归档时间（决策后 7 天自动归档） |

**emotion_tags**

| 列名 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 标签 ID |
| name | TEXT NOT NULL | 标签名称（2-6 字） |
| emoji | TEXT | 对应 emoji（可选） |
| is_predefined | INTEGER DEFAULT 1 | 1=系统预设, 0=用户自定义 |
| color_hex | TEXT | 标签颜色 |
| created_at | INTEGER | 创建时间（自定义标签） |

**cool_pool_item_tag_cross_ref** (多对多关联)

| 列名 | 类型 | 说明 |
|------|------|------|
| item_id | TEXT NOT NULL | FK → cool_pool_items.id |
| tag_id | TEXT NOT NULL | FK → emotion_tags.id |
| PRIMARY KEY (item_id, tag_id) | | |

**冷板凳数据（V2 扩展预留，V1 不实现 sync_queue、user_preferences 等表）**

### 5.3 关键索引

- `idx_items_status` ON cool_pool_items(status) → 列表筛选
- `idx_items_expires` ON cool_pool_items(expires_at) → 到期检查
- `idx_items_created` ON cool_pool_items(created_at DESC) → 列表排序

### 5.4 状态机

```
[cooling] ──用户决定购买──► [decided_buy] ──7天──► 归档
    │                           
    ├──用户放弃──► [decided_pass] ──7天──► 归档
    │                           
    └──到期──► [expired] ──用户决定购买──► [decided_buy]
                  │
                  ├──用户放弃──► [decided_pass]
                  │
                  └──再等等──► [cooling] (重置 expires_at = now + 72h)
```

## 6. 待确认决策点

以下问题需要主理人（team-lead）做出最终决策，会影响 V1.0 的交付范围和架构选择：

| # | 决策问题 | 方案 A | 方案 B | 建议 |
|---|---------|--------|--------|------|
| **DQ-01** | **离线优先程度**：V1 是否需要云同步？虽然 Firebase 被标记为 P2，但纯本地存储意味着用户换手机数据全丢。 | 纯本地 Room，无任何网络层，数据导出/导入作为唯一迁移途径 | 保留一个极简的 Room 备份到外部存储功能（JSON 文件），不做云同步 | 建议 **方案 B**，低成本保障数据安全感，且纯本地也能做 |
| **DQ-02** | **换算引擎的日薪数据**：日薪换算需要用户输入日薪，这个设置是否在 V1 做？如果做，放在哪里？ | 首次使用换算时弹出温和引导（可跳过），后续在个人中心可修改 | V1 不做日薪换算，仅做实物等价换算（奶茶/工作餐/健身卡等固定锚点） | 建议 **方案 B**，减少 V1 需要用户主动设置的步骤，实物锚点已足够有体感 |
| **DQ-03** | **情绪标签预设数量与内容**：原始 PRD 定义 8 个预设标签（开心/压力/无聊/被种草/奖励自己/习惯性消费/社交比较/其他），是否直接复用？ | 8 个预设标签全部实现，自定义上限 20 个 | 精简到 5 个核心标签（压力/被种草/奖励自己/社交比较/其他），V2 再扩展 | 建议 **方案 A**，情绪标签是核心数据壁垒，V1 一次性铺好基础 |
| **DQ-04** | **空状态 & 引导插画**：原 PRD 多处提及温暖手绘插画风格（小熊托气泡等），V1 是否产出定制插画？ | 使用自定义绘制的矢量插画（设计师产出） | 使用 Material Design 系统图标 + 温暖文案占位，V2 再配插画 | 建议 **方案 B**，V1 以功能验证为核心，插画可后续补充，减少设计资源依赖 |
| **DQ-05** | **P1 中哪个模块优先级最高**：如果时间允许只做一个 P1 模块，做哪个？ | 通知提醒（冷静期到期推送是核心闭环的必要触点） | 个人中心（数据导出/清除是隐私合规基础） | 建议 **方案 A**，通知是冷静期体验的最后一环，没有通知用户可能忘记回来决策，核心闭环断裂 |

## 附录：全局文案约束（V1 必遵）

| 禁止词汇 | 替代建议 |
|---------|---------|
| 省/省钱/节省 | 放下、缓冲 |
| 忍/控制/克制 | 先放一放、慢慢想 |
| 浪费/乱花/败家/剁手 | 不使用，不表达此语义 |
| 不该买/别买 | 你随时可以决定 |
| 冲动消费 | 情绪触发的想法 |
| 改善/问题/诊断 | 变化/模式/发现 |

> 所有换算结果必须后接"——供参考"。通知文案不使用感叹号。产品不说"贵""便宜""划算"，只呈现中性数字。
